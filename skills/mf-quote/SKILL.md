# MF Quote — 見積書作成スキル

ユーザーと対話しながら見積書を組み立てて、MoneyForwardクラウド請求書に下書きとして保存する。

## UX原則（必ず守る）

このスキルを使うのは**開発者ではない社員**も含まれる。以下を徹底する：

1. **専門用語は出さない** — 「API」「エンドポイント」「payload」「quote_id」「MCPツール」等の言葉はユーザーに一切見せない
2. **選択肢はボタンで出す（AskUserQuestion を使う）** — 「1/2/3 を入力してください」という番号入力ではなく、`AskUserQuestion` ツールを使ってタップで選べるUIを出す。タイピング負荷を最小化する
3. **まずテキストで見積を仕上げる** — MFへの反映は最後。テキスト上で金額・項目を確定してから
4. **反映は必ずユーザーの明示指示で** — 「反映して」「保存して」「MFに入れて」「登録して」など、またはAskUserQuestionの「反映する」ボタン押下で。自動で保存しない
5. **PDFダウンロードは頼まれた時だけ** — 「PDFで」「ダウンロードして」「PDFほしい」等の明示指示がない限り実行しない
6. **発行はしない** — 下書きのみ。発行・メール送付は MF 管理画面で人間がやる
7. **担当者名は見積書に書かない** — 会社名（株式会社CONTE）のみ

### AskUserQuestion を使うべきタイミング

以下のステップでは**必ず AskUserQuestion を使って選択肢ボタンで提示**する：

- Step 1: 作り方（おまかせ / 骨組みだけ / 複製）
- Step 2: 宛先が複数候補ある時（候補をボタン化）
- Step 2: 部門が複数ある時
- Step 4: 税別か税込かの確認
- Step 6: プレビュー後「反映する / 項目を編集する / 金額を変える / キャンセル」
- Step 7 後: 「PDFもダウンロードする？」（ここは聞かない方針だが、ユーザーが迷った時のみ）

自由入力が必要な項目（件名、金額目安、複製検索キーワード等）は普通にテキストで聞く。

## 実行フロー

### Step 0: 連携チェック（裏で静かに）

最初に内部的に `mf_me` を呼んで連携状態を確認する。
- 未連携なら：「まだMFと連携されていません。先に `/mf-setup` で連携してください」とだけ伝えて終了
- 連携済みなら：普通に始める。このチェックをユーザーに見せない

### Step 1: 作り方を聞く（AskUserQuestion で）

`AskUserQuestion` ツールを使ってボタン選択で出す。例：

```
question: "見積書を作ります。どの作り方にしますか？"
options:
  - label: "おまかせ"
    description: "件名と金額を教えてもらえれば、項目は私が考えます"
  - label: "骨組みだけ"
    description: "必要な項目だけ出して、金額は一緒に決めましょう"
  - label: "複製"
    description: "過去の見積を元に新しく作ります"
```

Skipボタンとフリー入力も自動で出るので、選択肢以外の指示があれば柔軟に対応。

### Step 2: 宛先を聞く（モード1・2・3共通）

```
宛先（お客様）はどちらですか？
```

（ここは自由入力 — 取引先名を入力してもらう。AskUserQuestionは使わない）

内部で取引先を検索。

- **候補0件**：
  ```
  MFに登録されていないお客様のようです。先にMF管理画面から取引先を登録してきてください。
  ```
- **候補複数**：`AskUserQuestion` を使ってボタン選択で候補を出す。例：
  ```
  question: "宛先はどちらですか？"
  options:
    - label: "〇〇市役所"
      description: "（主要顧客）"
    - label: "〇〇建具店"
      description: ""
    - label: "〇〇病院"
      description: ""
  ```
- **候補1件**：自動選択

部門がある取引先の場合、**部門複数なら `AskUserQuestion` でボタン選択**：

```
question: "どの部門宛にしますか？"
options:
  - label: "担当課"
  - label: "総務課"
  - label: "指定なし"
    description: "部門を指定せず会社全体宛てにする"
```

### Step 3: 件名を聞く

```
件名は？
```

### Step 4: 金額の目安を聞く

```
金額の目安はありますか？（「18万」「15万くらい」みたいに大雑把でOK、分からなければスキップできます）
```

**金額が出てきたら税別/税込を必ず確認**。これも `AskUserQuestion` でボタン選択：

```
question: "「18万」について、これはどちらの意味ですか？"
options:
  - label: "税別18万"
    description: "税別180,000円（税込198,000円）"
  - label: "税込18万"
    description: "税込180,000円（税別約163,636円）"
```

### Step 5: モード別に項目を組み立てる（テキスト上で）

**ここは MF にはまだ一切触らない。テキストでプレビューを見せるだけ。**

#### モード1（おまかせ）

件名のキーワード（配信／CM／ブランディング／Web／動画編集／撮影／AI／グラフィック等）を判定し、`references/quote-templates.md` をReadして該当テンプレートを参照。金額目安があれば税抜目標に合わせて各項目を配分。

#### モード2（骨組みだけ）

最低限の項目（カテゴリに応じて3〜5個）を金額空欄で出して、ユーザーと対話で金額を埋めていく。

#### モード3（複製）

```
複製したい過去の見積を探します。キーワードを教えてください（宛先・件名・案件名どれでもOK）：
```

検索結果を番号付きで提示して選んでもらう。選ばれたら**内部的に** `mf_get_quote` で中身を取得し、**テキスト上で**「この見積をベースにしますね」と提示。**まだ複製はしない**。

### Step 6: テキスト上でプレビュー & 調整

以下のような形で、MF に入れる前に全体を見せる：

```
こんな感じで組み立ててみました：

  宛先: 〇〇市役所（担当課）
  件名: 市民フォーラム 第3回 配信
  発行日: 2026-04-20
  有効期限: 2026-05-20
  税率: 10%

  項目:
    事前準備・会場下見              ¥18,000 × 1式
    機材費一式（カメラ等）          ¥40,000 × 1式
    機材運搬・設営・撤収費          ¥15,000 × 1式
    配信オペレーター費              ¥8,000 × 4時間 = ¥32,000
    音響費                          ¥15,000 × 1式
    アーカイブ編集・納品費          ¥10,000 × 1式
    進行管理費（打合せ込み）        ¥20,000 × 1式
                                   ──────────
                           小計     ¥150,000
                           消費税   ¥15,000
                           合計     ¥165,000 (税込)

この内容でよろしければ「反映して」と言ってください。
項目を足したい・消したい、金額を変えたい、件名を変えたい等あればそのまま教えてください。
```

**プレビューを見せた直後に、必ず `AskUserQuestion` で次アクションをボタン化**：

```
question: "この内容でMFに反映しますか？"
options:
  - label: "反映する"
    description: "この内容で下書きを保存します"
  - label: "項目を編集"
    description: "項目の追加・削除・金額変更"
  - label: "件名を変える"
  - label: "金額を変える"
  - label: "やめる"
    description: "作成をキャンセル"
```

ユーザーからの編集指示（「進行管理費を3万に」「音響費いらない」「件名を変えて」等）は、**テキスト上のプレビューを更新するだけ**で対応。MF にはまだ触らない。
編集後も毎回 `AskUserQuestion` で「反映する／さらに編集する／やめる」を聞く。

### Step 7: MF に反映する（ユーザーが明示指示したら）

ユーザーが以下のような言葉を出した時だけ、MF への保存を実行：
- 「反映して」「反映お願い」
- 「保存して」「保存」
- 「MFに入れて」「MFに登録して」「登録して」
- 「作って」「これで作って」
- 「OK」「OKです」（直前にプレビューを見せて「これでよければ反映してと言ってください」と案内した文脈で）

それ以外の曖昧な返事（「うーん」「いいかも」等）では反映しない。必ずもう一度確認する：

```
MFに反映していいですか？「反映して」と言ってもらえれば進めます。
```

反映処理:
- **モード1・2（新規作成）**: `mf_create_quote` で新規作成
- **モード3（複製）**: `mf_duplicate_quote` で複製→必要に応じて `mf_update_quote` / `mf_add_item` / `mf_delete_item` で編集

### Step 8: 反映完了を報告（URL を必ず提示）

保存が成功したら、返り値の `web_url` を使ってユーザーに報告：

```
✅ MFに下書きで保存しました。

  見積番号: 1234
  合計: ¥165,000（税込）
  MFで開く: https://invoice.moneyforward.com/quotes/1234

中身を確認して、問題なければ MF 管理画面から発行してください。
```

**PDFダウンロードは自動でしない**。ユーザーが「PDFで」「ダウンロードして」等と言った時だけ実行する。

### Step 9: PDFダウンロード（依頼された時だけ）

ユーザーが明示的に依頼したら `mf_download_pdf` を実行して、保存先を教える：

```
📄 PDF保存しました: ~/Downloads/見積書_〇〇市役所_フォーラム配信_1234.pdf
```

## 見積→請求書 変換フロー（v0.3.0〜）

ユーザーが**承認済みの見積を請求書に変換したい**と言った時の専用フロー。

### 起動トリガー（例）

- 「見積を請求書にして」「請求書化して」「請求書下書きを作って」
- 「見積#539を請求書にして」「〇〇社の見積を請求書に」
- 「お客様からOKが出たから請求書にして」
- 既存メールやSlackで「ご請求ください」「請求書発行お願いします」が来た文脈

### フロー

#### Step A: 対象見積の特定

quote_id が会話で明示されていない場合は `mf_search_quotes` で検索し、複数候補なら `AskUserQuestion` でボタン選択させる。

```
question: "どの見積を請求書にしますか？"
options:
  - label: "#539 伊藤豊嗣様 WEBサイト サーバー等 ¥81,400 (2026/04/24)"
  - label: "#396 伊藤豊嗣様 WEBサイト サーバー等 ¥81,400 (2025/04/23)"
```

#### Step B: 重複請求の確認

**必ず確認する**。MF API は同一見積から複数回 convert できる仕様か未検証のため、誤って二重請求書を作らないように：

```
question: "見積 #539（伊藤豊嗣様 WEBサイト サーバー等 ¥81,400）を請求書下書きに変換します。よろしいですか？"
options:
  - label: "請求書を作る"
    description: "下書きで保存。送付・入金登録はMF画面で人間が行います"
  - label: "やめる"
```

#### Step C: 変換実行

`mf_convert_quote_to_invoice(quote_id="...")` を呼ぶ。レスポンスの `id` / `billing_number` / `web_url` を使って結果報告：

```
✅ 請求書を下書きで作成しました。

  請求書番号: INV-001
  発行日: 2026-04-26
  支払期日: 2026-05-31
  合計: ¥81,400（税込）
  MFで開く: https://invoice.moneyforward.com/billings/xxxxx

中身を確認して、問題なければ MF 管理画面から発行・送付してください。
```

### 注意事項

- 請求書の発行・送付・入金登録はこのスキルでは行わない（**下書きまで**）
- PDFダウンロードはユーザーが明示的に依頼した時だけ。請求書PDFのダウンロードAPIは現状未実装（必要になったら billing 用の download tool を別途追加）
- 二重変換が起きないよう Step B の確認は省略しない
- 変換に失敗した場合（404・既に変換済み等）はAPIエラーをそのまま見せず、「この見積は変換できないようです。MF管理画面で確認してください」程度に丸める

## 詳細リファレンス

- **`references/api-endpoints.md`** — MFへの送信内容の構造（必須フィールド、品目構造、税区分）
- **`references/quote-templates.md`** — 件名カテゴリ別の項目テンプレート

必要になったタイミングでReadすればOK。最初から全部読む必要はない。

## エラー時の伝え方

ユーザーには**なぜ失敗したか**だけ分かる言葉で伝える。内部エラーコード（HTTP 401 等）はそのまま見せない。

| 内部で起きたこと | ユーザーへの伝え方 |
|---|---|
| `mf_me` が未認証 | 「まだMFと連携されていません。`/mf-setup` で連携してください」 |
| 401 / 認証エラー | 「MFとの連携が切れているようです。`/mf-setup` で再連携をお願いします」 |
| 422 / バリデーション | 「〇〇が足りない／おかしいようです。〇〇を教えてもらえますか？」 |
| 取引先0件 | 「MFに登録されていないお客様です。先にMF管理画面から取引先登録してください」 |
| ガードエラー（ロック済み等） | 「この見積はすでに発行済み／ロック済みで編集できません」 |

## 補足運用ルール

- 会社の定番は税込18万前後の定額案件（配信、CM、小規模ブランディング）
- 担当者名は見積書本体には入れない。MF管理画面上の「作成者・編集者」で自動追跡される
- 「下書きから発行」は基本的に代表が MF 管理画面でやる（または作成者が確認を取ってから発行）
